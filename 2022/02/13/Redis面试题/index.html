    Redis 基础篇

    1.说说你对 Redis 的了解？

    1. （基础简介）Redis 是由 C 语言编写，基于内存的非关系型数据库。一般用于架设在 Java 程序与数据库（MySQL）之间的缓存层来弥补应用程序多并发请求数据与数据库吞吐之间的差异，避免大量请求落入数据库从而起到保护数据库的作用。
    2. （高性能）由于是基于内存的数据库，所有它的性能十分优越，读操作能达到110000次/s，写操作能到达81000次/s。
    3. （可靠性）Redis可以使用 RDB 和 AOF 的形式将内存中的数据保存在硬盘里，可以避免发生断点或者机器故障造成数据丢失的问题。
    4. （可以用来做什么）Redis 目的是用来做缓存的，除了做缓存以外，还可以用来做分布式锁，甚至是消息队列。
    5. （多种数据类型）Redis 提供了多种数据类型来支持不同的业务场景，Redis 还支持事务、持久化、Lua 脚本、集群等。

    2.Redis 与 Memcached 的共同点与区别？

    共同点：

    1. 都是基于内存的数据库，一般都用来当做缓存使用。
    2. 都有键的过期策略。
    3. 两者的性能都非常高。

    不同点(区别)：

    1. （支持的数据类型角度）Redis 支持更丰富的数据类型(支持更复杂的应用场景)。 因为 Memcached 只支持最简单的 k/v 数据类型。Redis 不仅仅支持简单的 k/v 类型的数据，同时还提供 list，set，zset，hash 等数据结构的存储。
    2. （持久化方面）Redis 支持数据的持久化，可以将内存中的数据保持在磁盘中，重启的时候可以再次加载进行使用，而 Memecache 把数据全部存在内存之中，没有办法持久化，只能用于缓存使用，重启后数据全部丢失。
    3. （灾难恢复机制）因为 Redis 支持持久化，所以支持数据恢复机制，而 Memcached 不支持。
    4. （内存使用完的情况）Redis 在内存使用完的时候可以将内存中的数据持久化到硬盘里，而 Memcached 直接报异常。
    5. （集群）Redis 原生支持 cluster 模式，可以实现主从复制，读写分离。Memcached 没有原生的集群模式，需要依靠客户端来实现往集群中分片写入数据。
    6. （模型）Redis 6 之前，Redis 网络 IO 模型是单线程的多路 IO 复用模型，6.0版本开始使用多线程。而 Memcached 是多线程，非阻塞 IO 模式。
    7. （功能）Redis 支持事务、Lua 脚本、发布/订阅模式，而 Memcached 不支持。
    8. （过期键方面）Redis 支持定期删除和惰性删除，而 Memcached 只支持惰性删除。

    3.Redis 除了做缓存，还能做什么？

    • 消息队列

    Redis 自带的 list 数据结构可以作为一个简单的队列使用。也可以使用订阅/发布模型来做队列。Redis5.0 中增加的 Stream 类型的数据结构更加适合用来做消息队列。它比较类似于 Kafka，有主题和消费组的概念，支持消息持久化以及 ACK 机制。

    • 分布式锁

    4.Redis 运行快的原因？

    1. Redis 完全是在内存层面的操作，数据存在内存中，跟传统的磁盘文件数据存储相比，避免了从磁盘 IO 读取到内存这部分开销。
    2. Redis采用的是单线程 + IO 多路复用机制，单线程避免了线程切换的开销和竞争问题。减少了阻塞。IO 多路复用机制可以处理大量的并发 IO，而不用消耗太多 CPU/内存。
    3. 最最根本的是 Redis 是用 C 语言写的，直接跟操作系统交互，命令执行快的飞起。

    5.Redis 的优缺点？

    优点：

    • 数据结构丰富，除了支持 string 类型的 value 外还支持 hash、set、zset、list 等数据结构。

    • 读写性能优异， Redis能读的速度是 110000 次/s，写的速度是 81000 次/s。

    • 支持数据持久化，支持 AOF 和 RDB 两种持久化方式。

    • 支持事务，Redis 的所有操作都是原子性的，同时 Redis 还支持对几个操作合并后的原子性执行。

    • 支持主从复制，主机会自动将数据同步到从机，可以进行读写分离。

    缺点：

    • 数据库容量受到物理内存的限制，不能用作海量数据的高性能读写，因此 Redis 适合的场景主要局限在较小数据量的高性能

    操作和运算上。

    • Redis 不具备自动容错和恢复功能，主机从机的宕机都会导致前端部分读写请求失败，需要等待机器重启或者手动切换前端的 IP 才能恢复。

    • 主机宕机，宕机前有部分数据未能及时同步到从机，切换 IP 后还会引入数据不一致的问题，降低了系统的可用性。

    • Redis 较难支持在线扩容，在集群容量达到上限时在线扩容会变得很复杂。为避免这一问题，运维人员在系统上线时必须确保有足够的空间，这对资源造成了很大的浪费

    6.Redis4.0 之前为什么选择单线程+多路 IO 复用

    题外话：关于同步 IO、异步 IO、阻塞 IO、非阻塞 IO的联系与区别

    Redis 选择单线程的主要原因就是

    1⃣️ CPU不是 Redis 服务器的瓶颈。单线程也能执行的过来，引入多线程模型带来的性能提升并不能抵消多线程带来的开发成本和维护成本。

    2⃣️ 使用单线程模型能带来更好的维护性、方便开发和调试。

    3⃣️ 使用单线程模型也能并发的处理客户端的请求，因为采用的是I/O多路复用机制，可以统计等待多个连接发送的请求。

    注意⚠️：Redis 4.0 之后加入多线程也是在个别命令上加入的多线程，对于处理所有网络请求来说，还是单线程的！！

    关于多线程的引用：

    unlink命令（删除指定的key，不存在就跳过，跟del相同，但是相比于del不会阻塞，这会在另一个线程里进行。）

    flushall async 同步情况所有数据库，不会阻塞。

    flushdb async 同步清空当前数据库，不会阻塞。

    7.有 MySQL 了，为什么还需要用 Redis？

    两方面回答：高性能和高并发

    高性能：如果我们查询数据时，会直接在硬盘查询数据，速度比较慢，如果将用户所访问的数据存放在内存中，每次查询去内存中查找，速度快，因为没有和磁盘进行IO操作。如果数据库中对应的数据改变，同步缓存即可。

    高并发：Redis 能够承受的请求数是远远高于 MySQL 所能承受的请求数，如果我们在应用程序与数据库之间加入缓存层，能够提高系统的并发量。

    1.SDS(简单动态字符串)

    作用

    1. 保存字符串值
    2. 用作缓冲区( AOF 模块的 AOF 缓冲区、客户端状态的输入缓冲区)

    结构

    与 C 字符串的区别？

    1. 常数复杂度获取字符串的长度，时间复杂度为 O(1)

    2. 能有效杜绝缓冲区溢出

      C 字符串不仅不记录自身长度，而且没有分配足够多的内存的话，就会产生溢出。而 SDS 不会产生这样的问题，如果在插入的时候，先判断空间是不是满足，不满足先扩容，扩容完再放数据。

    3. 减少修改字符串时带来的内存重分配次数（空间预分配、惰性空间释放）

      空间预分配——上面提到的扩容也是有机制的

      • 长度小于 1M 的，程序最终会分 (len 属性的大小*2+1)字节。
      • 长度大于 1M 的，最后的空间是 (len 属性的大小+1M+1字节)

      惰性空间释放

      当空间缩小的时候，会直接记录到 free 字段里。

    4. 二进制安全的

      C 字符串只保存 ASCII 码，并且除了字符串的末尾，不能有空字符串，这些就导致了没办法存放图片、音频、视频等内容，而SDS 就随便放。它读取的是 len 的值，即使你有空字符串也继续读。使用的是 len 属性，而不是字符串里的空字符。

    3.字典

    作用

    1. Redis 底层数据库就是使用字典实现的
    2. 是哈希键的底层实现之一

    结构

    哈希算法

    • 计算要添加键值对中键的 hash 值
    • 然后再跟 sizemark 的值做计算，计算出索引值
    • 如果键冲突，就头插法插入链表

    rehash

    1⃣️为什么rehash？

    因为键值对逐渐增加或者减少，为了让负载因子维持在一个合理的范围，所以 rehash。

    2⃣️rehash 的前提

    1. 没有执行 BGSAVE 和 BGREWRITEAOF 命令 且 负载因子大于1
    2. 执行 BGSAVE 和 BGREWRITEAOF 命令 且 负载因子大于5

    主要是尽量避免在子进程存在期间进行 rehash 操作，最大的节约内存。

    3⃣️ rehash 步骤

    1. 为 ht[1] 分配空间

      • 如果执行的是扩展操作，那么ht[1]的大小为第一个大于等于 ht [0].used*2的2的n次方幂;（大于等于已经使用的空间的2倍，最接近的2的n次方）
      • 如果执行的是收缩操作，那么ht[1]的大小为第一个大于等于 ht [0].used的2的n次方幂。

    2. 将保存在 ht[0] 中的所有键值对 rehash 到 ht[1] 上面，rehash指的是重新计算键的哈希值和索引值，然后将键值对放置到ht[1] 哈希表的指定位置上。

    3. 当 ht[0] 包含的所有键值对都迁移到了 ht[1] 之后(ht [0]变为空表)，释放ht[0]，将 ht[1] 设置为 ht[0]，并在ht [1]新创建一个空白哈希表，为下一次rehash做准备。

    渐进式 rehash

    rehash 的操作不是一下子完成的，避免对服务器造成影响。

    步骤：

    1. 为 ht[1] 分配空间，让字典同时持有 ht[0] 和 ht[1] 两个哈希表。
    2. 在字典中维持一个索引计数器变量 rehashidx，并将它的值设置为 0 ，表示 rehash 工作正式开始。
    3. 在 rehash 进行期间，每次对字典执行添加、删除、查找或者更新操作时，程序除了执行指定的操作以外，还会顺带将ht[0]哈希表在 rehashidx 索引上的所有键值对 rehash 到 ht[1]，当 rehash 工作完成之后，程序将 rehashidx 属性的值增1。
    4. 随着字典操作的不断执行，最终在某个时间点上，ht[0]的所有键值对都会被rehash至ht[1],这时程序将rehashidx属性的值设为 -1，表示rehash操作已完成。

    注意⚠️：在 rehash 的时候查找，会先找 ht[0]，再找 ht[1]。如果插入的话就直接插在 ht[1] 中。

    4.跳跃表

    参考链接

    数组：二分查找时间 O(logN)，插入和删除都是 O(N)。

    链表：查找 O(N)，插入，删除 O(1)。

    因为查找效率太低了，我们引进了跳跃表，使他的查找时间复杂度尽可能的接近 O(logN)。

    作用

    • 有序集合键的底层实现之一 (zset)
    • 集群节点中用作内部数据结构

    时间复杂度

    平均 O(logN)，最坏 O(n)，因为实现起来更简单，并且效率和平衡树相媲美。

    结构

    跳跃表实现由zskiplist和zskiplistNode两个结构组成，其中zskiplist用于保存跳跃表信息（如表头节点、表尾节点、长度），而zskiplistNode则用于表示跳跃表节点。

    为什么不用二叉查找树？

    因为二叉查找树存在极端情况，会形成链表，不稳定。

    为什么不用红黑树？

    因为红黑树是二叉查找树的变形，查找、插入、删除的复杂度均为O(logN)，但是红黑树在插入、删除节点时都需要调整结构，花的时间很多。

    7. Redis 的常用指令？

    Redis 持久化篇

    RDB相关内容

    关键字：数据库的数量16、save、bgsave、RDB 文件的载入是自动的，RDB 自动达到一定条件自动保存

    AOF相关内容

    AOF的持久化功能实现分为追加(append)、文件写入、文件同步。三个步骤，AOF的持久化效率和安全性主要是受flushAppendOnlyFile()函数。而flushAppendOnlyFile()函数依赖于appendfsync选项的值，具体看问题6。

    参考《Redis 设计与实现》P140

    AOF重写：新文件不会包含浪费空间的命令，之前是多个命令，现在是取出键值对，用命令来记录，最后保存到AOF文件中。并且AOF重写的时候，还有AOF重写缓冲区，在AOF开始重写的时候，这个缓冲区建立，然后重写阶段的其他来自客户端的命令结果都保存到这个缓冲区里，等AOF重写完，然后在执行写AOF重写缓冲区这些。

    1. 讲一下 Redis 中的持久化机制

    RDB

    RDB 持久化方式是将 Redis 在内存中的数据库记录按照指定时间写入磁盘，实际操作过程是 fork(BGSAVE命令)一个子进程，先将数据写入临时文件，写入成功后，在替换之前的文件，是压缩的二进制文件。

    AOF

    AOF 持久化是以日志的形式记录服务器所处理的每一个增加、删除、修改操作。不会记录查询操作，以文本的方式记录，可以打开文件看到详细的操作记录。

    2. 说一下 RDB 和 AOF 的优缺点？

    RDB 的优势

    • 适合大规模数据恢复（个人理解为启动效率高）
    • 恢复速度快
    • 可以对已经存在的 RDB 文件进行压缩（LZF算法压缩），节省磁盘空间。
    • 性能最大化，对于持久化的时候，他唯一做的就是 fork 出子进程，之后再由子进程完成这些持久化工作，这样就能极大的避免主进程 IO 操作了。

    RDB 的缺点

    • 最后的一次修改可能会丢失（因为它是周期性做备份的）
    • 虽然 Redis 在 fork 时使用了写时拷贝技术，但是如果数据庞大时还是比较消耗性能。
    • fork 的时候，内存中的数据被克隆了一份，大致2倍的膨胀性需要考虑。

    AOF 的优点

    • 可以让数据更安全，丢失数据概率更低。
    • 可读的日志文本，可以处理误操作。
    • AOF 对日志文件的写入是追加的方式，所以即使在追加的时候出现宕机，之前的操作也是没什么问题的，而且 Redis 还有工具可以解决问题。

    AOF 的缺点

    • 比起 RDB 更占空间（因为不仅仅保存数据还保存操作的记录）
    • 恢复速度比较慢（因为需要一条条执行完指令）
    • 存在 bug，可能造成不能恢复。

    3. RDB 和 AOF 选哪一个？

    这个要具体情况具体分析

    如果只是拿 Redis 做内存缓存的话，那么可以都不用。

    如果对于数据很敏感，可以使用 AOF，但是不建议单独使用，可以也打开 RDB。

    如果对数据不敏感而且还要进行保存，那么仅使用 RDB 进行存储。

    4.SAVE 和 BGSAVE 命令的区别？

    首先，这两个命令都用于生成RDB文件，最后都执行的是rdbSave()命令

    SAVE 命令：会阻塞Redis服务器进程，直到RDB文件创建为止，在阻塞期间，服务器不能处流任何请求。

    BGSAVE 命令：会派生出一个子进程，然后由子进程负责创建RDB文件。

    5.BGSAVE 命令执行期间，关于SAVE、BGSAVE、BGREWRITEAOF命令

    如果在执行BGSAVE命令，SAVE命令会被拒绝，服务器禁止同时执行两个 rdbSave命令。同时如果是BGSAVE命令，也会拒绝，因为两个BGSAVE也会产生竞争条件。

    如果要执行的是BGREWRITEAOF命令，并且在执行BGSAVE命令，那么此时BGREWRITEAOF命令会被延迟到BGSAVE命令执行完。反过来也一样。

    6.appendfsync参数

    appendfsync always：始终同步，每次Redis的写入都会立刻记入日志；性能较差但数据完整性比较好

    appendfsync everysec：每秒同步，每秒记入日志一次，如果宕机，本秒的数据可能丢失。

    appendfsync no：redis不主动进行同步，把同步时机交给操作系统。

    Redis 事务篇

    1. Redis 事务的概念？

    Redis 事务的本质就是通过 MULTI、EXEC、WATCH、DISCARD等一组命令的集合，事务支持一次执行多条命令，一个事务中所有的命令都会被序列化，在事务执行的过程中，会按照顺序串型化执行队列中的命令，其他客户端提交的命令请求不会插入到事务执行命令中去。

    MULTI(multi)：命令开启一个事务，执行该命令后，一系列操作会被放入事务中。

    EXEC(exec)：命令执行，一系列操作完成。

    WATCH(watch)：监视某一个键，如果在事务执行之前，被监视的键被修改了，那么这个事务就无法执行。

    DISCARD(discard)：清空队列，放弃执行事务。

    2. Redis 事务的三个阶段、三个特性

    三阶段：

    1. 开启事务(MULTI)
    2. 命令入队
    3. 执行事务(EXEC) / 撤销事务(DISCARD)

    三特性：

    1. 单独的隔离操作：事务的所有命令都会序列化，按顺序执行，事务在执行的过程中，不会被其他客户端发送过来的命令请求所打断。
    2. 没有隔离级别的概念，因为每一条命令都攒着，一直到执行 EXEC 命令。
    3. 不保证原子性：因为如果一个事务里面的一条命令执行失败，其后面的命令依旧执行，没有回滚。

    注意⚠️：在组队（入队）的时候出错，执行时整个的所有队列都会被取消。

    但是在执行过程中出错的时候，那么就不保存原子性了，其他都执行，错误的就不执行。

    3.Redis 事务保证原子性吗？支持回滚吗？

    在 Redis 中，单条命令是原子性支持的，但是事务不保证原子性的，且没有回滚。

    Redis 集群篇

    主从模式

    主从复制包括全量复制，增量复制两种。一般当 slave 第一次启动连接 master，或者认为是第一次连接，就采用全量复制。以后再次发生更新，就会触发增量复制。

    主从复制的原理：

    • 当从服务器连上主服务器的时候，从服务器向主服务器发送数据同步消息
    • 主服务器连接到从服务器发送过来的同步消息后，把主服务器数据进行持久化RDB文件，把 RDB 文件发送给从服务器，然后，从服务器拿到文件后进行读取。
    • 每次主服务器进行写操作了，这个时候主服务器主的进行数据同步。

    哨兵模式

    解决主从模式中一旦主节点出现故障而不能提供服务，需要人工将从节点晋升为主节点的问题。

    哨兵模式，由一个或多个 Sentinel 实例组成的 Sentinel 系统，它可以监视所有的 Redis 主节点和从节点，并在被监视的主节点进入下线状态时，自动将下线主服务器属下的某个从节点升级为新的主节点。但是，一个哨兵进程对 Redis 节点进行监控，就可能会出现问题（单点问题），因此，可以使用多个哨兵来进行监控 Redis 节点，并且各个哨兵之间还会进行监控。

    哨兵的三个作用：

    1. 发送命令，等待Redis服务器（包括主服务器和从服务器）返回监控其运行状态；

    2. 哨兵监测到主节点宕机，会自动将从节点切换成主节点，然后通过发布订阅模式通知其他的从节点，修改配置文件，让它们切换主机；

    3. 哨兵之间还会相互监控，从而达到高可用。

    故障切换的过程

    假设主服务器宕机，哨兵1先检测到这个结果，系统并不会马上进行 failover 过程，仅仅是哨兵1主观的认为主服务器不可用，这个现象成为主观下线。当后面的哨兵也检测到主服务器不可用，并且数量达到一定值时，那么哨兵之间就会进行一次投票，投票的结果由一个哨兵发起，进行 failover 操作。切换成功后，就会通过发布订阅模式，让各个哨兵把自己监控的从服务器实现切换主机，这个过程称为客观下线。这样对于客户端而言，一切都是透明的。

    哨兵的工作模式如下：

    1. 每个 Sentinel 以每秒钟一次的频率向它所知的 Master，Slave 以及其他 Sentinel 实例发送一个 PING 命令。

    2. 如果一个实例（instance）距离最后一次有效回复 PING 命令的时间超过 down-after-milliseconds 选项所指定的值， 则这个实例会被 Sentinel 标记为主观下线。

    3. 如果一个 Master 被标记为主观下线，则正在监视这个 Master 的所有 Sentinel 要以每秒一次的频率确认Master的确进入了主观下线状态。

    4. 当有足够数量的 Sentinel（大于等于配置文件指定的值）在指定的时间范围内确认Master的确进入了主观下线状态， 则Master会被标记为客观下线。

    5. 在一般情况下， 每个 Sentinel 会以每10秒一次的频率向它已知的所有 Master，Slave 发送 INFO 命令。

    6. 当Master被 Sentinel 标记为客观下线时，Sentinel 向下线的 Master 的所有 Slave 发送 INFO 命令的频率会从 10 秒一次改为每秒一次。

    7. 若没有足够数量的 Sentinel 同意 Master 已经下线， Master 的客观下线状态就会被移除；若 Master 重新向 Sentinel 的 PING 命令返回有效回复， Master 的主观下线状态就会被移除。

    Redis 内存篇

    1. Redis 的过期键的删除策略？

    可供选择的过期键的删除策略有三种

    • 定时删除

    在设置键的同时，创建了一个定时器，让定时器在键过期的时候，执行删除操作。

    优点：对内存友好，能保证过期键被快速删除，释放过期键所占用的内存。

    缺点：对CPU不友好，在过期键比较多的情况下，删除过期键会占用相当多的CPU时间。会对服务器的相应时间和吞吐量造成影响。

    • 惰性删除

    程序只会在取出键时才会对键进行过期检查，保证删除过期键的操作只会在非做不可的情况下进行，删除的目标仅限于当前处理的键，不会处理其他过期的键。

    优点：对CPU友好，由于在处理的时候不会处理其他的键，减少了CPU在删除过期键的时间。

    缺点：对内存不友好，如果一个键过期，而这个键又不会被使用，就一直就保留在数据库中，所占用的内存就不会被释放。

    • 定期删除

    定期删除就是：每隔一段时间就对数据库进行一次检查，删除里面过期的键。

    难点：难以确定删除操作执行的时长和频率，如果执行的时间太久或者频率太频繁，就会退化成定时删除，如果执行的时间太短或者频率太少，就会退化成惰性删除。

    定时删除和定期删除属于主动删除，惰性删除属于被动删除。

    Redis 只用了两种：惰性删除和定期删除。

    2.如何保证 Redis 中都是热点数据？

    例如：MySQL 里面有2000W数据，Redis 中只存20W数据。

    正确设置内存淘汰策略，当 Redis 内存数据集大小上升到一定大小的时候，就会实施数据淘汰策略，从而剩下的就是热点数据。

    3. Redis 的淘汰策略有哪些？

    8种

    • volatile-lru（least recently used）：从已设置过期时间的数据集中挑选最近最少使用的数据淘汰。

    • volatile-ttl：从已设置过期时间的数据集中挑选将要过期的数据淘汰。

      理解：也就是移除那些 TTL（Time To Live） 值最小的 key，即那些最近要过期的 key。

    • volatile-random：从已设置过期时间的数据集中任意选择数据淘汰。

    • allkeys-lru：当内存不足以容纳新写入数据时，在键空间中，移除最近最少使用的 key（这个是最常用的）

    • allkeys-random：从数据集中任意选择数据淘汰。

    • no-eviction（禁止驱逐）：不进行移除。也就是说当内存不足以容纳新写入数据时，新写入操作会报错。

    4.0 版本后增加两种：

    • volatile-lfu：从已设置过期时间的数据集中挑选最不经常使用的数据淘汰。

    • allkeys-lfu：当内存不足以容纳新写入数据时，在键空间中，移除最不经常使用的 key。

    1. Redis 如何实现分布式锁？

    参考链接：https://www.zhihu.com/question/300767410/answer/1749442787

    问题出现的原因

    单机中，我们使用本地锁就可以避免多个操作共同操作同一个数据的问题，但是在分布式系统中，很容易就是多个客户端修改同一个数据，又因为读取和更新不是一个操作，所以原来的本地锁就派不上用场，这个时候就就需要分布式锁，它的作用就是保证多台客户端在执行某一段代码时保证只有一台客户端执行。

    如果实现分布式锁，需要满足的条件
    1. 互斥性，任何时刻，保证只有一个客户端持有锁。
    2. 不能出现死锁，即使崩溃也要按时释放。
    3. 解锁和释放必须是同一个客户端。
    目前可以实现的方式
    • 使用 MySQL 实现分布式锁
    • 基于缓存（Redis）也是我们要讲的（性能最高）
    • 基于 Zookeeper实现分布式锁（最可靠）
    情况一

    使用setnx key 1设置锁，使用del key删除锁。

    出现的问题：

    • 处理业务时，业务异常没及时释放锁。
    • 进程挂了，没释放锁。
    情况二

    对情况一改进，先执行 setnx key 1,再执行 expire key 10,设置过期时间。

    出现的问题：只执行完了第一条命令，却没来的及执行第二条命令。这时候和情况一相同。

    情况三

    对情况二改进，执行set key 1 ex 10 nx，这样就是原子操作了，也解决了死锁问题

    出现的两个问题：

    1. 锁过期：客户端A加锁成功，开始操作共享资源，但是操作共享资源还没完成，已经超过加锁设置的时间，锁自动释放。
    2. 释放别人的锁：客户端B获得客户端A释放的锁后，加锁成功，开始操作共享资源，此时客户端A执行完后，释放了客户端B的锁。
    情况四

    解决上面第一个问题就是给锁续期。在 Redisson 框架实现分布式锁的思路，就使用 watchDog 机制实现锁的续期。当加锁成功后，同时开启守护线程，默认有效期是30秒，每隔10秒就会给锁续期到30秒，只要持有锁的客户端没有宕机，就能保证一直持有锁，直到业务代码执行完毕由客户端自己解锁，如果宕机了自然就在有效期失效后自动解锁。

    解决第二个问题就是使用UUID，set key UUID ex 10 nx，

    情况五

    在删除锁的情况下的，需要判断锁是不是自己持有的。此操作不是原子性操作，假设查询到锁的UUID是自己的，但是又正好刚刚过期，就会是其他线程抢占锁，而此时原来的线程还是会清理锁。

    解决方式：判断和清理需要用到 Lua 脚本。

    情况六

    之前分析的都是单个 Redis 实例中可能产生的问题，如果在集群中呢？

    场景：

    1. 客户端 A 在主库上执行 SET 命令，加锁成功
    2. 此时，主库异常宕机，SET 命令还未同步到从库上(主从复制是异步的)
    3. 从库被哨兵提升为新主库，这个锁在新的主库上，丢失了!

    解决方式 ： RedLock

    Redlock 的方案基于 2 个前提：

    1. 不再需要部署从库和哨兵实例，只部署主库
    2. 但主库要部署多个，官方推荐至少 5 个实例

    步骤：

    1. 客户端先获取「当前时间戳T1」
    2. 客户端依次向这5个 Redis 实例发起加锁请求(用前面讲到的 SET 命令)，且每个请求会设置超时时间(毫秒级， 要远小于锁的有效时间)，如果某一个实例加锁失败(包括网络超时、锁被其它人持有等各种异常情况)，就立即向下一个Redis实例申请加锁。
    3. 如果客户端从 >= 3个(大多数)以上 Redis 实例加锁成功，则再次获取「当前时间戳T2」，如果T2- T1 <锁的过期时间，此时，认为客户端加锁成功，否则认为加锁失败。
    4. 加锁成功，去操作共享资源(例如修改MySQL某一行，或发起一个API请求)
    5. 加锁失败，向「全部节点」发起释放锁请求(前面讲到的Lua脚本释放锁)

    2. Redis 如何实现消息队列？

    参考链接-@Kaito

    参考链接-JavaKeeper

    不建议使用 Redis 做消息队列，因为 Redis 设计的初衷是为了做缓存。如果做消息队列的话也可以，注要有以下三种方式。

    三种方式：

    • List 队列
    • 发布/订阅模型
    • Stream 趋于成熟的队列

    其实还有一种是专业的消息队列，比如 kafka、RabbitMQ。

    先介绍一下什么是消息队列：

    消息队列 是指利用 高效可靠 的 消息传递机制 进行与平台无关的 数据交流，并基于数据通信来进行分布式系统的集成。通过提供 消息传递 和 消息排队 模型，它可以在 分布式环境 下提供 应用解耦、弹性伸缩、冗余存储、流量削峰、异步通信、数据同步 等等功能，其作为 分布式系统架构 中的一个重要组件，有着举足轻重的地位。

    List 队列

    1. 模型简单，一边生产者生产消息，一边消费者拉取消息.

      存在的问题：没有消息就一直拉取，这个时候就会导致 CPU 空转。如果要是让拉取消息睡眠一会，就有可能导致新消息延迟消费。

    2. 改进：使用 BRPOP/BLPOP ，这里的 B 是阻塞的意思。没有消息就阻塞，有了消息就能消费。

      存在的问题：

      • 不支持多个消费者重复消费，消费者拉取消息后，List 上就被删除了，无法被其他消费者消费。
      • 消息丢失，消费者拉取消息后宕机了，那这条消息就丢失了。

    解决方式看发布/订阅模型

    发布/订阅模式

    可以解决了 List 队列不可重复消费的问题，多个消费者订阅同一个生产者，当生产者有消息时，通知消费者，并且一个消费者可以订阅多个生产者。但是存在的问题就是丢数据。

    发生的场景：1. 消费者下线。2. Redis 宕机。3.消息堆积

    我们可以看到，发布/订阅模型不存储数据，只是转发给消费者。没有数据持久化的能力。

    如果消费者下线了，那再次上线只能接受新的消息。

    如果 Redis 宕机，Pub/Sub 的数据也会全部丢失。

    消费者必须先订阅队列，生产者才能发布消息，否则消息会丢失。

    如果生产者生产速度比较快，对于List来说，可能就是链表过长，内存持续增长，如果是 Pub/Sub 模式，因为消费者每次订阅一个队列，都会给消费者一个缓冲区（也就是一块内存），当生产者发布消息的时候，就会把消息写到订阅这个的消费者的缓冲区，之后消费者在缓冲区读取消息，但是消费者缓冲区是有限制的，如果超过了缓冲区，那就会把这个消费者踢下线。也就导致消费失败，丢失数据。

    发布/订阅的优缺点：

    1. 支持发布/订阅。支持多组生产者，消费者处理消息。（优点）

    2. 消费者下线，消息丢失。

    3. 不支持数据的持久化，Redis 宕机，数据丢失。

    4. 消息堆积，缓冲区溢出，消费者被踢，数据丢失。

    5. 从缓冲区取走数据，也就在 Redis 缓冲区中删除了，消费者异常，无法重新消费。

    我们希望的是：

    1. 支持阻塞式拉取消息
    2. 支持发布/订阅模式
    3. 消费失败，可以重新消费。
    4. 实例宕机，消息不丢失，数据可持久化。
    5. 消息可堆积。

    Stream

    Stream 使用 XADD 和 XREAD 完成生产消费，也支持阻塞式拉取消息，也支持发布/订阅模式，当消费完成后，会执行 XACK 命令告知 Redis，Redis 就会标记为处理完成。

    Stream 和其他数据类型一样，写操作都会写入到 RDB 和 AOF 中。

    对于消息堆积，两个解决方法，1. 生产者限流。2. 丢弃消息：中间件丢弃消息，只保留固定长度新消息。所以，当消息积压的时候，还是会丢失消息。有小瑕疵，如果继续对比，那就是与专业的消息队列对比了。

    3. 缓存穿透、缓存击穿、缓存雪崩

    3.1 缓存穿透

    既不存在于内存数据库 Redis，也不存在于数据库 MySQL。

    问题理解：缓存穿透就是大量请求的数据key并不在缓存和数据库中，导致直接请求到了数据库，从而可能压垮数据库。举个例子，黑客故意制造我们缓存中不存在的 Key，而发起大量请求，就会可能导致我们数据库宕机。

    解决方案：

    • 首先是做好参数校验，应该对无效请求直接抛出异常。

    • 对无效key缓存：如果这个数据不在数据库中，那么直接存在缓存中设置，最长不超过5分钟，但是此种方法不适合大量请求不存在的key的情况。

    • 设置可访问名单（待定）

    使用bitmaps类型定义一个可以访问的名单，名单id作为bitmaps的偏移量，每次访问和bitmap里面的id进行比较，如果访问id不在bitmaps里面，进行拦截，不允许访问。

    • 采用布隆过滤器

    它实际上是一个很长的二进制向量(位图)和一系列随机映射函数(哈希函数)。布隆过滤器可以用于检索一个元素是否在一个集合中。它的优点是空间效率和查询时间都远远超过一般的算法，缺点是有一定的误识别率和删除困难。将所有可能存在的数据哈希到一个足够大的 bitmaps 中，一个一定不存在的数据会被这个 bitmaps 拦截掉，从而避免了对底层存储系统的查询压力。

    • 进行实时监控

    当发现Redis的命中率开始急速降低，需要排查访问对象和访问的数据，和运维人员配合，可以设置黑名单限制服务。

    3.2 缓存击穿（针对一个Key）

    问题理解：缓存中存在一个刚刚过期的key，还没有被清除，同时有大量请求去访问这个 key 时，同时去数据库中读取数据。可能会把数据库压垮。

    解决方案：

    • 预先设置热门数据：在 redis 高峰访问之前，把一些热门数据提前存入到 redis 里面，加大这些热门数据 key 的时长
    • 实时调整：现场监控哪些数据热门，实时调整 key 的过期时长
    • 使用锁。这样可以保证只有一个用户请求去访问数据库并缓存到Redis中，其他请求可以通过再次访问Redis即可。

    3.3 缓存雪崩（针对大量Key）

    问题理解：缓存中的大量数据 key 到过期时间，而查询数据量巨大，可能引起数据库压力过大或者宕机。

    解决方案：

    • 将缓存失效时间设置随机，防止大量数据同时过期现象的发生。
    • 设置热点数据永不过期。
    • 如果缓存数据是分布式部署，可以将数据均匀分布在不同的缓存数据库中。

    4. 如何解决数据不一致性？（Redis - MySQL）

    不理解？点我看原文！

    数据的不一致性：是指假如一个数据访问者同时读取 Redis 和数据库，在一段时间里发现这两个数据不一致。

    解决的办法有两种：

    第一种：先删除缓存，在更新数据库。

    第二种：先更新数据库，在删除缓存。

    从两个方面考虑，并发和第二步操作失败的情况。

    对于并发角度：

    如果有两个线程要并发读写数据，就可能造成数据不一致。

    • 先删除缓存，再更新数据库
    • 先更新数据库，在删除缓存

    但是这种情况概率很低，需要满足以下条件：

    • 缓存刚刚失效
    • 读请求与写请求并发
    • 更新数据库+删除缓存时间要比读数据库+写缓存时间短，但是更新数据库都要加锁，通常时间更长。

    所以，这种情况是可以保证数据一致性的。

    对于第二步操作失败的情况：

    无论是上面两种情况的哪一种，只要是第二步失败，就会导致不一致性，因此，解决问题的关键就是让第二步执行成功。我们可以采取 重试 的方法，但是立即重试会大概率失败，并且也不知道设置多少重试次数才合理。所以才有异步重试。把重试请求写入到消息队列里。让消息队列保证第二步执行成功。

    另外一种方案就是：订阅数据库变更日志，再操作缓存。只要是修改了数据库，那就有相应的变更日志，拿到具体操作（用阿里的canal），再去删除缓存。这种情况的优点就是(1).无需考虑消息队列失败的情况。(2).自动投递下层队列。

    总结：如果想要保证数据库与缓存的一致性，推荐采用 更新数据库+再删除缓存的方案，配合消息队列或订阅变更日志的方式来做。

    1.布隆过滤器？

    参考文章

    Redis 笔记

    应对缓存穿透问题，我们可以使用布隆过滤器。布隆过滤器是什么呢？

    布隆过滤器是一种占用空间很小的数据结构，它由一个很长的二进制向量和一组Hash映射函数组成，它用于检索一个元素是否在一个集合中，空间效率和查询时间都比一般的算法要好的多，缺点是有一定的误识别率和删除困难。

    布隆过滤器原理：

    假设我们有个集合A，A中有n个元素。利用k个哈希散列函数，将A中的每个元素映射到一个长度为a位的数组B中的不同位置上，这些位置上的二进制数均设置为1。如果待检查的元素，经过这k个哈希散列函数的映射后，发现其k个位置上的二进制数全部为1，这个元素很可能属于集合A，反之，一定不属于集合A。

    2.Redis 底层使用的是什么协议？

    RESP 是 Redis 客户端与 Redis 服务器相互通信时使用的一个协议， 全称 REdis Serialization Protocol ，即 redis 串行协议，通俗易懂，也表明了 redis 的特点，串行化(单线程)。

    Redis 的作者在设计 RESP 协议，制定了下面的三条原则：

    1. 实现要简单
    2. 对计算机来说，解析速度快
    3. 对人类来说，可读性强
